Quick update regarding the Acunetix UAT setup delay.

I completed end-to-end troubleshooting today across the network, firewall, application, and server layers to isolate the issue. Network connectivity and firewall access over HTTPS/443 were validated successfully with the network team, however the application itself was not coming up.

Further investigation on the UAT server showed that the Acunetix services are configured against paths on a secondary application volume (E:) which currently does not exist on the UAT clone. Comparing against PROD confirmed that the Acunetix application/runtime components reside on E: in PROD, while the current UAT setup only contains the OS/application base volume (C:).

As a result, the Acunetix services fail to start since the configured E:\ paths are missing on UAT.

Before proceeding further, I wanted to check with you whether you would prefer that I go ahead and raise a request to provision/restore the E: volume on UAT, or if you would like to discuss with the OS team first given the additional storage requirement.
